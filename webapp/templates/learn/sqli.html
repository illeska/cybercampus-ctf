Injection SQL (SQLi)

Comprendre et prévenir les attaques sur les bases de données

Critique 📖 15 min ⚡ Débutant

🔍 Qu'est-ce qu'une injection SQL ?

L'injection SQL est une technique d'attaque qui permet à un utilisateur malveillant d'injecter du code SQL dans une requête de base de données. Cette vulnérabilité survient lorsque les données fournies par l'utilisateur sont directement intégrées dans une requête SQL sans validation ni échappement approprié.

Les conséquences peuvent être catastrophiques : contournement de l'authentification, vol de données sensibles, modification ou suppression de données, et même prise de contrôle totale du serveur dans certains cas.

💡
Le saviez-vous ?

L'injection SQL figure dans le Top 3 du classement OWASP depuis plus de 15 ans. Malgré sa dangerosité connue, elle reste l'une des vulnérabilités les plus exploitées sur le web.

⚙️ Comment fonctionne une attaque SQLi ?

Prenons un exemple concret d'un formulaire de connexion vulnérable :

❌ Code vulnérable Python / Flask 
username = request.form['username']
password = request.form['password']

# Danger : Concaténation directe dans la requête SQL
query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
cursor.execute(query)

user = cursor.fetchone()
if user:
    login_user(user)

Ce code semble fonctionnel, mais il est extrêmement dangereux. Voici pourquoi :

Scénario d'attaque :

Un attaquant entre les valeurs suivantes dans le formulaire :

admin' OR '1'='1' --
(n'importe quoi)

La requête SQL devient alors :

SELECT * FROM users WHERE username='admin' OR '1'='1' --' AND password='...'

Analyse de l'attaque :

  • 'admin' : Tentative de connexion en tant qu'admin
  • OR '1'='1' : Condition toujours vraie qui court-circuite la vérification
  • -- : Commentaire SQL qui ignore tout ce qui suit (le mot de passe)

Résultat : L'attaquant se connecte en tant qu'admin sans connaître le mot de passe !

Autres types d'attaques :

🔓 Bypass d'authentification

Se connecter sans identifiants valides

admin' --

📂 Extraction de données

Récupérer des informations sensibles

' UNION SELECT password FROM users --

💣 Modification de données

Altérer ou supprimer des enregistrements

'; DROP TABLE users; --

🛡️ Comment se protéger ?

Heureusement, il existe plusieurs méthodes efficaces pour prévenir les injections SQL :

1

Requêtes préparées (Prepared Statements)

La méthode la plus sûre : séparez complètement la structure SQL des données utilisateur.

✅ Code sécurisé Python / SQLite 
username = request.form['username']
password = request.form['password']

# Les ? sont des placeholders - les données sont échappées automatiquement
query = "SELECT * FROM users WHERE username=? AND password=?"
cursor.execute(query, (username, password))

user = cursor.fetchone()
Pourquoi c'est sûr : Les paramètres sont traités comme des données pures, jamais comme du code SQL. Impossible d'injecter des commandes.
2

ORM (Object-Relational Mapping)

Utilisez un framework qui abstrait les requêtes SQL et gère la sécurité automatiquement.

✅ Code sécurisé Python / SQLAlchemy 
from models import User

username = request.form['username']
password = request.form['password']

# SQLAlchemy échappe automatiquement les valeurs
user = User.query.filter_by(
    username=username,
    password=password
).first()

if user:
    login_user(user)
3

Validation et filtrage des entrées

Vérifiez toujours les données utilisateur avant de les utiliser :

  • Définissez des types de données attendus (int, email, etc.)
  • Limitez la longueur des champs
  • Utilisez des expressions régulières pour valider le format
  • Créez une liste blanche de caractères autorisés
4

Principe du moindre privilège

Limitez les permissions du compte de base de données utilisé par votre application :

  • N'utilisez jamais un compte root ou admin
  • Accordez uniquement les permissions nécessaires (SELECT, INSERT, UPDATE)
  • Interdisez DROP, CREATE, ALTER sur les comptes applicatifs
  • Séparez les comptes par environnement (dev, prod)

📝 Résumé des bonnes pratiques

✅ Utilisez toujours des requêtes préparées ou un ORM
✅ Ne concaténez JAMAIS directement les entrées utilisateur dans du SQL
✅ Validez et filtrez toutes les données côté serveur
✅ Limitez les privilèges de vos comptes de base de données
✅ Loggez et surveillez les tentatives d'injection
✅ Gardez vos frameworks et bibliothèques à jour
